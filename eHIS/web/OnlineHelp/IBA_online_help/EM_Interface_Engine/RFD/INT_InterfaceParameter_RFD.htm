Enter the following details:

 

  1. Enable Interface – Select the check box to enable the interface. This in turn triggers the CSC events (required to be interfaced) to write the event information in to the control log table.

 

General

  1. Setup Master Code Mapping – Select the check box to enable the master code mapping. This enables you to define the HL7 interface cross-reference setup between CSC and interfaced application(s). Selecting this check box enables you to define the code mapping in the ‘Data Element Cross Reference’ function.

  2. Enable Event Filtering – Select the check box to enable event filtering. Enabling this parameter allows you to configure the export rules based on interfaced applications. If this check box is not selected, the filter rules cannot be defined in the ‘Filtering and Scheduling’ function.

  3. Temporary Purge Interval (days) – Enter the temporary purge interval in days. The records (events, messages, audit logs) are maintained in the main table for the specified number of days, after which the records are moved to the respective backup tables.  

  4. Permanent Purge Interval (days) – Enter the permanent purge interval in days. Records (events, messages, audit logs) older than the specified number of days are deleted from the respective back up tables. To avoid any data (history) loss, it is recommended to take a back up of the backup tables respectively.  

Consider the Temporary Purge interval as 10 days and the Permanent Purge interval as 100 days. The events, messages, audit logs ( if any) are stored in the main table for 10 days and after the 10th day, the records are moved to the respective backup tables. The records are stored in the backup tables for 100 days (permanent purge interval) and records that are older than 100 days are deleted from the backup tables.

 

Communication Process Settings

  1. Database IP address – Enter the Database IP address in valid format. Communication process uses this parameter setup for connecting to the database. Value specified in the parameter is similar to the values that are defined for ODBC connection. Valid format for this parameter is as follows: Database IP Address: Database Listener Port:Database SID

  2. Idle Time (min) – Enter the idle time (in minutes). This parameter indicates as to how long the HL7 interface communication process (outbound or inbound) can be idle, that is, without processing any HL7 interface communication messages. When the idle time of the communication process reaches this specified limit, the background process automatically closes the connection. In such cases, the administrator has to restart the communication process to reestablish the HL7 interface between CSC and interfaced applications.

 

Facility

The current facility is displayed with the log setup.

  1. Audit – Select the check box to enable the audit log function. Audit log displays the details as to when the various messages were rebuilt, replayed, etc.

  2. Debug – Select the check box to enable the debug log function. The debug log function is also referred to as the trace log. Enabling this log is not recommended because the system creates thousands of lines of log for one transaction and enabling this log would slow down the system.

  3. Outbound – Select the check box to enable the outbound log function. By enabling this parameter, all unsolicited messages are stored in a log file.

Log file is written in the directory as per the definition made for ‘Dir’ entry under ‘Reports’ section in function ‘Workstation Based Parameters’. The file is written using the file naming nomenclature as ‘outbound####.log’; where #### refers the communication Process ID that has processed the message.

  1. Inbound - Select the check box to enable the inbound log function. By enabling this parameter, all unsolicited messages are stored in a log file.

Log file is written in the directory as per the definition made for “Dir” entry under ‘Reports’ section in function ‘Workstation Based Parameters’. The file is written using the file naming nomenclature as ‘inbound####.log’; where #### refers the communication process ID that has processed the message.
